# Jira Service-Desk Monitor (`/jira`)

A live dashboard inside LL-Gatus that mirrors the Jira Cloud service desk the same
way the main page mirrors the network: KPIs, breakdowns, a created-vs-resolved
trend, per-ticket SLA countdowns, a Kanban board, and a click-through drill-down
for any ticket. It updates in real time over SSE, just like the WAN board.

Reach it from the Jira icon in the top-right of the header, or at `/jira`.

---

## What it shows

- **Project tabs** — one per configured project. Long Lewis has two:
  - **LLSM** — *IT Service Management* (the JSM help desk; has SLAs and queues).
  - **LLIP** — *IT Projects* (has a Jira agile/Kanban board, id `5`).
- **KPI rail** — open tickets, unassigned, SLA breached, average resolution time,
  created today, resolved today.
- **Breakdowns** — open tickets by priority and by type (compact segmented bars)
  plus a status-chip row, and a 14-day created-vs-resolved sparkline.
- **Open tickets** — a sortable list (click any column header) or a **Board** view
  (Kanban columns by status). Toggle with the List/Board buttons.
  - Tickets sort by **SLA urgency** by default (most overdue first).
  - Each ticket shows a **live SLA countdown** that ticks every second and turns
    amber under 1h, red under 15m, and "overdue" in red once breached.
  - New tickets **flash** when they arrive.
- **Drill-down** — click a ticket to slide open a panel with its summary, type,
  status, priority, assignee, reporter, labels, all SLA metrics with remaining
  time, the full rendered description, and recent comments. It refreshes itself
  every 15s while open, and deep-links to the ticket in Jira.

## Architecture

Gatus polls Jira itself (we hold the API token) on a background goroutine, caches
a snapshot in memory, and pushes it to browsers over SSE. Ticket detail is fetched
on demand when a drawer opens.

```
Jira Cloud REST  <--poll (30s)--  jira poller (Go)  --snapshot-->  in-memory store
                                                          |
                                          SSE broadcast   v
   browser  <--/api/v1/jira/live (SSE)-----------  api handlers  <--/metrics, /issue/:key
```

### Backend files

| File | Purpose |
|------|---------|
| `jira/jira.go` | Config, HTTP client (Basic auth), the poll loop, metric aggregation, the in-memory store, and the SSE broadcaster (`Subscribe`/`Unsubscribe`/`broadcast`). |
| `jira/issue.go` | `FetchIssue()` — on-demand single-ticket detail (description via `renderedFields`, SLA cycles, recent comments). |
| `jira/demo.go` | Synthetic snapshot for `JIRA_DEMO=1` (preview the UI without live creds). |
| `api/jira.go` | HTTP handlers: `GetJiraMetrics`, `GetJiraIssue`, `JiraLive` (SSE). |
| `api/api.go` | Registers the three routes; **excludes `/api/v1/jira/live` from the compress middleware** (SSE must stream unbuffered). |
| `main.go` | Calls `jira.StartPoller()` from `start()`. |

### Frontend files

| File | Purpose |
|------|---------|
| `web/app/src/views/JiraDetails.vue` | The `/jira` page: tabs, KPI rail, breakdowns, sparkline, sortable list + board, live SSE + SLA countdowns. |
| `web/app/src/components/JiraTicketPanel.vue` | The drill-down slide-over (teleported, sanitizes description HTML with DOMPurify, live-refreshes). |
| `web/app/src/App.vue` | The header Jira button (see the *header icon gotcha* below). |
| `web/app/src/router/index.js` | Adds the `/jira` route. |
| `web/app/src/assets/jira.png` | The Jira icon (inlined into the bundle at build). |
| `web/static/*` | The built bundle (regenerated by `npm run build`; embedded into the Go binary). |

### HTTP endpoints (added)

| Method | Path | Returns |
|--------|------|---------|
| GET | `/api/v1/jira/metrics` | The latest cached snapshot (all projects). Always 200; `configured`/`ok` say whether it's set up and healthy. |
| GET | `/api/v1/jira/issue/:key` | On-demand detail for one ticket. |
| GET | `/api/v1/jira/live` | SSE stream; emits a full snapshot on every poll. |

### Jira Cloud endpoints used

The classic `/rest/api/3/search` was removed by Atlassian in 2025, so we use the
current ones:

- `GET /rest/api/3/myself` — **auth check** (see gotcha below).
- `GET /rest/api/3/project/{key}` — project display name.
- `POST /rest/api/3/search/approximate-count` — fast counts.
- `POST /rest/api/3/search/jql` — issue lists (paginated via `nextPageToken`).
- `GET /rest/servicedeskapi/request/{key}/sla` — SLA cycles (breach time, paused, remaining).
- `GET /rest/api/3/issue/{key}?expand=renderedFields` — drill-down (rendered description HTML).
- `GET /rest/api/3/issue/{key}/comment` — recent comments.

## Configuration

All environment-driven (in `.env`, injected into the `gatus` container via
`env_file`). Nothing polls until `JIRA_BASE_URL`, `JIRA_EMAIL` and
`JIRA_API_TOKEN` are all set.

| Variable | Required | Default | Notes |
|----------|----------|---------|-------|
| `JIRA_BASE_URL` | yes | — | e.g. `https://longlewis.atlassian.net` (no trailing slash). |
| `JIRA_EMAIL` | yes | — | The Atlassian account email that **owns the API token**. Must match exactly. |
| `JIRA_API_TOKEN` | yes | — | Atlassian API token. Used as HTTP Basic `email:token`. |
| `JIRA_PROJECTS` | no | `LLSM,LLIP` | Comma-separated project keys to monitor. |
| `JIRA_DEMO` | no | `0` | `1` serves synthetic data (badged "demo data"). |
| `JIRA_POLL_SECONDS` | no | `30` | How often the poller refreshes (min 15). |
| `JIRA_TREND_DAYS` | no | `14` | Days in the created-vs-resolved trend/sparkline. |
| `JIRA_MAX_ISSUES` | no | `500` | Pagination cap per query. |
| `JIRA_SLA_MAX` | no | `60` | Max open tickets per project checked for SLA (0 disables SLA). |
| `JIRA_SLA_PROJECTS` | no | first project | Which projects to pull SLA data for (service desks). |
| `JIRA_TYPE_INCIDENT` / `_SERVICE_REQUEST` / `_PROBLEM` | no | Incident / Service Request / Problem | Override if your issue-type names are custom-labeled. |

### Getting a working token

1. At **id.atlassian.com → Security → API tokens**, create a token while logged
   in as the account that can see the projects.
2. Put the token in `JIRA_API_TOKEN` and that account's **exact** email in
   `JIRA_EMAIL`.
3. Set `JIRA_DEMO=0`, then recreate the container (`docker compose up -d`).

## Build & deploy

The Dockerfile **only compiles the Go binary**; it does not run `npm`. The Vue app
is pre-built into `web/static/` (which is `go:embed`'d). So a frontend change is a
two-step build:

```bash
cd web/app && npm run build        # regenerates web/static/
cd ../.. && docker compose up -d --build gatus
```

On the prod box the flow is git-based: commit + push to `origin/main`, then
`./update.sh` (force-syncs, rebuilds, verifies the running build SHA).

### Merging the Jira env into an existing `.env`

`.env` is gitignored and lives only on each box. To add the Jira keys to a prod
`.env` without disturbing the existing (master) values, use `env-merge.sh`:

```bash
./env-merge.sh .env.incoming .env   # target (.env) is master; only missing keys are appended
```

It backs up the target, keeps every existing value, and appends only keys the
target doesn't already define. Idempotent and safe to re-run.

## Gotchas & troubleshooting

**All zeros but "healthy" / no error.** Bad Jira credentials do **not** return 401
on the search endpoints — Jira treats the caller as *anonymous* and returns empty
results. That's why the poller verifies auth with `/rest/api/3/myself` first and
reports "authentication failed" instead of a misleading empty board. The usual
cause is the email: it must be the exact Atlassian account email (e.g.
`colby.west@longlewis.net`, not `colbywest@…`).

**Header Jira button renders as a solid white box.** `config.yaml`'s
`ui.custom-css` includes `html.dark header img { filter: brightness(0) invert(1) }`
(intended for the wordmark logo). It repaints *any* `<img>` inside `<header>`
white. The Jira button therefore renders the icon as a **background-image `<span>`**
(`.jira-ico` in `App.vue`), not an `<img>`, so the rule can't touch it.

**Live updates don't stream / indicator stays "updated Xs ago".** SSE responses
must bypass gzip. `/api/v1/jira/live` is excluded from the compress middleware in
`api/api.go`; if you add another SSE route, exclude it there too.

**SLA countdown looks frozen.** SLA clocks only run within the service desk's
business-hours calendar. Outside those hours a ticket's `slaActive` is false and
the UI shows the (static) remaining time rather than ticking. Overdue tickets show
"overdue" in red regardless.

**Preview without live Jira.** Set `JIRA_DEMO=1` and recreate the container to see
the full UI populated with synthetic LLSM/LLIP data (clearly badged "demo data").

## Which "board" to pull from

- **LLSM** is a JSM **service desk** (id `2`) — it uses **queues**
  (`/rest/servicedeskapi/servicedesk/2/queue`), not an agile board.
- **LLIP** has an agile **Kanban board** ("LLIP board", id `5`,
  `/rest/agile/1.0/board/5`).

The built-in **Board** view groups the live tickets by status (a Kanban view over
the same data). Pulling a project's *actual* board columns/queues 1:1 is not wired
up yet but is a straightforward addition on top of the two APIs above.
